# Отчёт по этапу 2

## 1. Цель этапа

Подтвердить, что выбранный на первом этапе стек `Vosk + RHVoice` может быть реализован как автономный офлайн-модуль для системы оповещения, поддерживающий русскоязычный TTS/STT, единый сценарий управления и демонстрацию ключевых команд.

## 2. Что реализовано

- TTS-адаптер RHVoice и REST API для синтеза.
- STT-адаптер Vosk и REST API для распознавания WAV / микрофона.
- Оркестратор, связывающий STT, логику команд, MQTT и TTS.
- Монолитный режим, GUI-режим и микросервисный режим через Docker Compose.
- Мониторинг через Prometheus и Grafana.
- Автотесты и CLI для замеров.

## 3. Архитектура

Компоненты:

1. `STT Service` принимает WAV и возвращает распознанный текст.
2. `TTS Service` принимает текст и воспроизводит его или сохраняет в WAV.
3. `Orchestrator Service` определяет команду, публикует MQTT-событие и формирует голосовой ответ.
4. `GUI / CLI` используются для локальной демонстрации без Docker.

Интеграционный поток:

1. Аудио поступает через микрофон или HTTP.
2. Vosk распознаёт текст.
3. Реестр команд определяет целевое действие.
4. Оркестратор публикует MQTT-команду.
5. RHVoice озвучивает подтверждение или ответ.

## 4. API и интерфейсы

| Компонент | Метод | Назначение |
| --- | --- | --- |
| STT | `POST /stt/recognize` | Распознавание WAV |
| TTS | `POST /tts/generate` | Синтез текста |
| Orchestrator | `POST /process` | Полный STT -> command -> TTS цикл |
| Все сервисы | `GET /health` | Проверка доступности |

Авторизация: `Authorization: Bearer <VOICE_API_TOKEN>` или `X-API-Key`.

## 5. Минимальные требования к ресурсам

- CPU: от 4 логических ядер
- RAM: от 4 ГБ
- Диск: от 8 ГБ
- GPU: не требуется
- Сеть: нужна только для администрирования, основной контур работает офлайн

Рекомендация для демонстрации: Linux-машина или Raspberry Pi 4/5 с облегчённой русской моделью Vosk и установленным RHVoice.

## 6. Сценарии использования в ГГС

- голосовая команда локального оператора (`включи свет`, `выключи свет`);
- запрос статуса (`какая температура`);
- выдача предзаписанных или динамически синтезированных сообщений;
- пилотная интеграция с виртуальным складом через MQTT-темы `factory/...`.

Подробные пошаговые прогоны описаны в `docs/demo_scenarios.md`.

## 7. Тестирование и измерения

Автоматические тесты:

- проверка маршрутизации команд;
- защита API и rate limiting;
- ограничения путей сохранения;
- поведение GUI при голосовой остановке;
- корректная потоковая передача аудио в orchestrator.

Измерения:

- CLI `python -m app.cli.benchmark` формирует `reports/stage2_measurements.json`;
- фиксируются latency, CPU load и peak memory для маршрутизации и response pipeline;
- при наличии Vosk-модели и RHVoice выполняются реальные STT/TTS-замеры.

## 8. Сравнение с расчётными ожиданиями этапа 1

- выбранный стек подтверждён как работоспособный в офлайн-контуре;
- интеграция не требует GPU;
- микросервисный режим масштабируется по сервисам STT и TTS независимо;
- фактические программные зависимости и требования к железу укладываются в ожидаемый промышленный low-cost контур.

## 9. Вывод

Критерии второго этапа закрыты:

- рабочий прототип TTS и STT есть;
- компоненты интегрированы в монолит и микросервисы;
- есть CLI, GUI и API для демонстрации;
- подготовлены тесты, метрики, отчёт и экономическое обоснование.
